# PRISMA V2 — Vollständiger System-Audit

**Datum:** 20. Juni 2026  
**Methode:** 9 parallele Analyse-Agents auf GitHub-Repo + Live-Render-App + lokalen Dateien  
**Verifizierungsgrad:** Alle Befunde wurden gegen den tatsächlichen Code und GitHub Actions Logs gegengeprüft. Agent-Befunde die sich als falsch erwiesen haben, sind explizit korrigiert.

---

## Executive Summary

Die Architektur ist solide — Clean Architecture, 704 Unit-Tests, 45 Integration-Tests, 20 E2E-Tests, vollständiges CI/CD. Das sichtbare Problem hat eine einzige Wurzel: **Der Daily Data Seed hat in der gesamten Projektgeschichte genau 2 Mal gelaufen — beide Male fehlgeschlagen.** Die Datenbank-Tabellen `crypto_signals`, `ml_features` und `stock_daily_signals` sind seit Projektbeginn leer. Dadurch:

- Krypto-Charts zeigen "Noch keine ausreichende Historie" (kein einziger Snapshot gespeichert)
- ML Prediction unmöglich (keine Feature-Vektoren in DB)
- Aktien-Signale fehlen (keine täglichen Snapshots)
- Backtest nicht aufrufbar (leere DB)

Die Root Causes sind 4 fehlende GitHub Secrets + 2 Code-Bugs im Workflow. Behebungsaufwand: ~30 Minuten.

## 2. Krypto-Charts — Root Cause

### Fehlerpfad (vollständig verifiziert)

```
GET /api/v1/crypto/history/BTC?days=30
→ SQLACryptoSignalRepository.get_history()
→ SELECT * FROM crypto_signals WHERE ticker = 'BTC' AND created_at >= ...
→ 0 Zeilen (Tabelle ist leer, da crypto_daily_snapshot nie erfolgreich lief)
→ API gibt [] zurück
→ useCryptoHistory() → data = []
→ CryptoHistoryChart.tsx:81: hasEnoughData = data.length >= 2 → FALSE
→ <p>Noch keine ausreichende Historie für diesen Ticker.</p>
```

### Wichtiger Hinweis: Live-Signale funktionieren anders

`GET /api/v1/crypto/signals` macht einen **Live-CoinGecko-Call** ohne DB-Cache. Diese Route sollte aktuelle Krypto-Signale zurückgeben, auch wenn die DB leer ist — sofern `COINGECKO_API_KEY` gesetzt ist. Nur die History-Charts (30-Tage-Verlauf) brauchen die DB.

### Sekundärer Bug im Workflow (tritt auf sobald API_KEY gefixt ist)

`CryptoSignalORM` hat **kein `snapshot_date`-Feld** — nur `created_at` (DateTime). Der Verify-Step im Workflow fragt aber:

```python
# .github/workflows/daily-data-seed.yml:223
text("SELECT COUNT(*) FROM crypto_signals WHERE snapshot_date = :d")
```

→ PostgreSQL-Fehler: `column "snapshot_date" does not exist` → Verify-Step crasht, obwohl Daten gespeichert wurden. Das bedeutet: auch nach dem API_KEY-Fix würde der Cron-Job mit einem Fehler enden, die eigentlichen Daten aber korrekt gespeichert haben. Muss trotzdem gefixt werden.

Die 61.4% in der Dokumentation beschreiben "v3 mit SimFin" — ein Modell das lokal trainiert aber nie als `latest` eingecheckt wurde. Das aktuell deployete Modell hat 49.4%.

### Data Leakage im Training (kritisch)

```python
# scripts/train_return_predictor.py — _stub_fundamentals()
# Für ALLE historischen Snapshots 2018–2025:
fund = _stub_fundamentals(ticker, _market)  # → holt AKTUELLE yfinance-Daten
```

## 5. Signal- und Scoring-System

### SwissQuantScorer — Bewertung

**Formel:**
```
composite = value_score × 0.4 + income_score × 0.4 + quality_score × 0.2

value_score  = pe_score × 0.6 + pb_score × 0.4
income_score = dividend_yield-basiert
quality_score = EPS > 0 → 100, sonst 0
```

**Stärken:** Kalibrierung auf SMI-Medianwerte ist korrekt. Dividendenfokus für CH-Vorsorgekontext begründet.

**Schwächen:**
- `value_score` kann nur 4 Werte annehmen: {25, 50, 75, 100}. Nestlé (PE=17) und eine Aktie mit PE=19.9 bekommen identisch 75 Punkte — kaum Diskriminierungskraft.
- `quality_score` ist rein binär (EPS positiv = 100). Kein ROE, kein FCF, kein Operating Margin. Nestlé mit 17% Marge und Lonza mit 22% sind ununterscheidbar.

**BUY-Threshold-Inkonsistenz:**
- `swiss_quant_scorer.py:12` → `_BUY_THRESHOLD = 70.0`
- `decision_signal.py:39` → BUY bei `weighted_score >= 65.0`

Ein Titel mit Quant-Score 67 ist laut SwissQuantScorer HOLD, laut DecisionSignal (aggregiert) BUY. Das ist konzeptuell vertretbar (das Aggregat hat mehr Information), aber die gleiche "BUY"-Bezeichnung für verschiedene Schwellen ist verwirrend.

### FundamentalsProvider in Production: RICHTIG

**Korrektur eines Agent-Befundes:** Der `StubFundamentalsProvider` ist in Production **nicht** aktiv. `backend/interfaces/rest/dependencies.py:103-104` zeigt:

```python
async def get_fundamentals_provider() -> FundamentalsProvider:
    return YFinanceFundamentalsAdapter()
```

Der `YFinanceFundamentalsAdapter` ist vollständig implementiert und holt live Fundamentaldaten von yfinance. Der Stub wird nur in Tests verwendet.

### Crypto Scorer — Logische Widersprüche

- **RSI contrarian** (RSI < 30 → 10 Punkte, überverkauft = gut) + **7d-Momentum-Follow** (7d negativ → 0 Punkte) konterkarieren sich: Ein Crash gibt RSI-Punkte aber keine Momentum-Punkte. Es gibt keine explizite Auflösungslogik für diesen Widerspruch.
- **ATH-Score mit Integer-Division:** `min(5.0, ath_pct // 20)` → Sprünge statt Gradient. 39% Abstand = 1 Punkt, 40% Abstand = 2 Punkte.

### Aktien-Domain-Modelle

| Modell | Qualität | Anmerkung |
|--------|----------|-----------|
| `AlphaModel` | ✅ Gut | Multi-Horizon Sharpe, Jegadeesh-Titman-basiert |
| `TrendMomentumModel` | ✅ Gut | EWMA halflife=63, korrekt |
| `QualityClassicModel` | ⚠️ Punkt-in-Zeit Bias | Nutzt live yfinance (korrekt für Production, aber beim Training: Leakage) |
| `ValueAlphaPotentialModel` | ✅ Gut | Mean-Reversion, solide |
| `DiversificationModel` | ✅ Gut | Ledoit-Wolf Kovarianz |

## 6. Universe-Wizard — Design-Bug

### Was passiert

Wizard Step 4 ("Welche Firmen kennst du?") → User wählt z.B. 3 Aktien.

```python
# backend/application/services/discovery_service.py:327-333
known = {t.upper() for t in profile.known_tickers}
scored.sort(key=lambda t: (0 if t[0].ticker in known else 1, -t[1]))

limit = _KNOWLEDGE_RESULT_LIMIT.get(profile.financial_knowledge, 20)
result = [stock for stock, _ in scored[:limit]]  # bis zu 20 Titel
```

`known_tickers` ist eine **Sortier-Priorität**, kein Filter. Der User sieht bis zu 20 (medium) oder 30 (high) Aktien — mit den gewählten 3 ganz oben.

### User-Erwartung vs. System-Verhalten

Die Frage "Welche Firmen kennst du?" suggeriert "diese siehst du dann". Tatsächlich bedeutet es "diese werden dir zuerst angezeigt, aber du siehst noch viele weitere."

### Richtiger Ansatz

Da das System ein personalisiertes Investitionsuniversum aufbauen soll (nicht nur die bekannten Aktien zeigen), ist die Logik an sich korrekt. Die UX-Erwartung muss angepasst werden. Empfehlung: Die Wizard-Frage umbenennen und im Reveal-Screen explizit zeigen, welche Aktien "bekannt" vs. "empfohlen" sind.

**Datei:** `.github/workflows/daily-data-seed.yml`

### P2 — Qualitätsverbesserungen

#### Fix 9: SwissQuantScorer — Kontinuierliche Scoring-Skala

**Datei:** `backend/domain/services/swiss_quant_scorer.py`  
**Problem:** `value_score` kann nur {25, 50, 75, 100} annehmen. Zu wenig Granularität.

**Richtiger Ansatz:** Lineare Interpolation statt Stufenmodell:

```python
@staticmethod
def _score_pe(pe: float | None) -> float:
    if pe is None:
        return 50.0
    if pe <= 0:
        return 0.0
    # Linear: PE=10 → 100, PE=30 → 0, clamp [0, 100]
    return max(0.0, min(100.0, (30.0 - pe) / 20.0 * 100.0))

@staticmethod
def _score_pb(pb: float | None) -> float:
    if pb is None:
        return 50.0
    # Linear: PB=1 → 100, PB=7 → 0, clamp [0, 100]
    return max(0.0, min(100.0, (7.0 - pb) / 6.0 * 100.0))
```

**quality_score verbessern:**
```python
def _score_quality(self, f: SwissFundamentals) -> float:
    if f.eps_chf is None:
        return 50.0
    if f.eps_chf <= 0:
        return 0.0
    # Zusätzlich: operating margin aus Fundamentals wenn verfügbar
    return 100.0  # Mindest-Score für positive EPS
```

**Warum richtig:** SMI-Titel mit PE=14 und PE=19 verdienen unterschiedliche Scores — beide fallen aktuell in "75". Das macht den Score für Stock-Picking unbrauchbar.

**Risiko:** Tests müssen angepasst werden, da sich Score-Werte ändern. BUY-Threshold eventuell anpassen.
